BÁO CÁO TIẾN ĐỘ PHÁT TRIỂN HỆ THỐNG CHATBOT RAG

1. Mục tiêu

Xây dựng hệ thống chatbot có khả năng:

  • Đọc và hiểu tài liệu PDF
  • Trả lời câu hỏi dựa trên nội dung tài liệu
  • Giảm thiểu sai lệch của mô hình ngôn ngữ
  • Triển khai chạy local (không phụ thuộc API bên ngoài)

Giải pháp sử dụng: Retrieval-Augmented Generation (RAG)

2. Kiến trúc hệ thống

Pipeline:

PDF → Chunking → Embedding → FAISS → Retrieval → LLM → Answer

2.1 Thành phần chính

  • Document Loader: đọc file PDF
  • Text Splitter: chia nhỏ văn bản
  • Embedding Model: chuyển text thành vector
  • Vector Store (FAISS): lưu trữ vector
  • Retriever: tìm context liên quan
  • LLM: sinh câu trả lời cuối cùng

3. Chi tiết triển khai

3.1 File ingest.py (Xây dựng vector database)

Pipeline xử lý dữ liệu gồm các bước:

  • Load toàn bộ PDF trong thư mục data/
  • Sử dụng RecursiveCharacterTextSplitter:
    • chunk_size = 500
    • chunk_overlap = 50
  • Tạo embedding bằng model nomic-embed-text
  • Lưu vào FAISS tại vectorstores/db_faiss

Lý do thiết kế:

  • Chunking nhỏ (500): giúp tăng độ chính xác retrieval
  • Overlap (50): tránh mất thông tin giữa các đoạn
  • FAISS: tốc độ truy vấn nhanh, phù hợp hệ thống local

Kết quả:

  • Dữ liệu được vector hóa hoàn chỉnh
  • Cho phép truy vấn semantic search hiệu quả

3.2 File chatbot.py (Hệ thống hỏi đáp)

Sử dụng pipeline RAG thông qua LangChain RetrievalQA

  • Load vector database từ FAISS
  • Retriever với top_k = 3
  • Sử dụng LLM: llama3
  • Áp dụng custom prompt

Thiết kế prompt:

  • Chỉ được trả lời dựa trên context
  • Nếu không có thông tin → trả lời "không biết"

Cấu hình quan trọng:

  • temperature = 0.1 → giảm độ ngẫu nhiên
  • Giới hạn context → giảm nhiễu

Ý nghĩa:

  • Giảm sai lệch
  • Tăng độ tin cậy của câu trả lời
  • Đảm bảo output bám sát tài liệu

4. Lý do lựa chọn model

4.1 Embedding Model: nomic-embed-text

  • Hiệu quả trong semantic search
  • Chạy local được
  • Tương thích tốt với FAISS và LangChain

Không chọn các model lớn hơn vì:

  • Tốn tài nguyên
  • Không cần thiết với bài toán tài liệu nội bộ

4.2 LLM: llama3

  • Hiểu ngữ cảnh tốt
  • Trả lời chính xác hơn các model nhỏ
  • Ổn định khi kết hợp với RAG

5. Kết quả đạt được

  • Xây dựng hệ thống RAG
  • Truy vấn context chính xác từ tài liệu
  • LLM trả lời dựa trên dữ liệu thực

Hành vi hệ thống:

  • Câu hỏi có trong tài liệu → trả lời chính xác
  • Câu hỏi ngoài tài liệu → trả lời "không biết"

6. Hạn chế hiện tại

  • Embedding chưa tối ưu cho tiếng Việt
  • Chưa xử lý tốt tài liệu dạng scan (OCR chưa tích hợp)
  • Hệ thống đôi khi trả lời bằng tiếng Anh do ảnh hưởng từ model
  • Tốc độ phản hồi còn chậm khi dữ liệu lớn
  • Chưa có cơ chế hội thoại nhiều lượt (multi-turn)

7.Thắc mắc và hướng phát triển

  • Làm sao để chọn giá trị top_k tối ưu cho retrieval
  • Làm sao để hỗ trợ hội thoại nhiều lượt, trong đó câu hỏi sau có thể kế thừa ngữ cảnh từ câu trước
  • Làm sao để tối ưu tốc độ hệ thống (giảm thời gian embedding, retrieval và inference của LLM)
  • Có nên sử dụng embedding từ HuggingFace (ví dụ các model dangvantuan/vietnamese-document-embedding, keepitreal/vietnamese-sbert ) để cải thiện chất lượng semantic search cho tiếng Việt
  • Khi có tài liệu mới, cần cập nhật vector database như thế nào (thêm/xóa/sửa embedding) mà không phải rebuild toàn bộ hệ thống
  • Làm sao tối ưu truy vấn lặp lại (repeated queries), ví dụ sử dụng caching để tránh phải thực hiện retrieval và inference lại từ đầu
